## ADR-009 — 任务管理的 4 个小决策（确认）

**日期**：2026-06-07

把 SPEC §5.5 里几个容易被未来的自己质疑的小决定打包记下来，避免回头反悔。

### 9.1 打卡 × 任务关联：5 秒自动确认，宁可错关联

**选项**：默认确认 / 默认不关联，等用户确认

**决定**：默认确认。5 秒非模态提示，不操作就写入 `task_events`。

**理由**：打断成本 > 错关联成本。错的事后任务详情页一键删 task_events 行就行，但每次都问会让你越来越懒得打卡。

**推翻条件**：用户反馈"经常错关联且懒得清"→ 改为默认不关联。

### 9.2 任务已用时估算：每条打卡按 60 分钟算

**选项**：60 分钟/打卡 / 两次打卡之间的真实间隔 / 用户手动填

**决定**：60 分钟/打卡（粗略估算，可在设置里改为 30/45/60/90/120）。

**理由**：
- 打卡间隔是真实时长的话，会被中断（吃饭、回消息、刷手机）夸大
- 让用户手动填工时表，App 立刻死
- 60 分钟和打卡间隔一致，符合直觉
- 反正只用作"哪些任务花了我比较多时间"的相对比较，绝对值不重要

**推翻条件**：积累 1 个月数据后发现某些场景误差极大且让你产生错误判断→改算法。

### 9.3 任务面板的位置：独立窗口

**选项**：独立窗口 / 回顾页加一个任务 tab

**决定**：独立窗口（菜单栏点"任务"打开），不进回顾页。

**理由**：
- 任务是**操作**频繁的（改状态、加任务、删任务），回顾页是**阅读**为主——两种模式不该共用一个窗口
- 任务窗口要常驻打开（边工作边看），回顾页是偶尔打开的
- 但任务窗口里可以跳到回顾页的相关时间段（链接式跳转，不是合一个窗口）

**推翻条件**：实际用了之后发现"我需要边看任务边看时间轴"，就再合并。

### 9.4 状态只有 5 个，不加 blocked / waiting

**决定**：`pending / in_progress / done / cancelled / archived`，就这 5 个。

**理由**：
- blocked / waiting 是项目管理工具的状态，单人记录用不上——"在等谁回复"自己心里清楚，用 tag `#等回复` 就够了
- 状态越多，状态机越复杂；状态机越复杂，UI 越乱；UI 越乱，用得越少
- 简单不是缺功能，是设计选择

**推翻条件**：发现 70% 的 pending 任务实际是"在等别人"，且这种区分有价值→加 blocked。

## ADR-008 — 集成任务管理（深集成），独立 tasks 表

**日期**：2026-06-07

**背景**：用户提出 todo 任务管理也应该接入——"计划做什么"和"实际做了什么"是一对镜像，放一起价值最大。用户当前没在用任何任务工具，没有迁移成本。

**选项**：
1. 浅集成 —— 在 events 表加 `type='todo'`，只当事件记，不做任务列表 UI
2. **深集成 —— 独立 tasks 表 + task_events 关联表 + 完整列表 UI + 打卡自动关联**
3. 不做，建议用户用外部工具（Things / Todoist），App 只做读取

**决定**：选 2。任务独立成 `tasks` 表，通过 `task_events` 关联表和 events 双向关联；打卡时 LLM 自动尝试关联到 pending/in_progress 任务，5 秒确认窗口；任务列表是独立功能页；任务的"创建/开始/完成"三个时刻同时落到 events 表，保证回顾时间轴看得到。

**理由**：
- 用户没在用任务工具——这是个干净的设计起点，不用兼容
- 浅集成（方案 1）拿不到核心价值"任务总耗时"——这是和 Todoist 唯一的真区别
- 任务有生命周期（pending → in_progress → done），不只是某个时刻的事，结构上不该塞进万物皆 event 的 JSON 字段
- 任务字段查询频繁（"所有 pending 任务"、"按优先级排"），存 events.structured JSON 里性能差
- 关联表设计允许一个任务关联多次打卡，天然支持"耗时累计"
- 不做的关键决定：循环规则、深层子任务、看板、依赖、提醒推送——这些是"另一个 Todoist"的方向，不是我们要去的地方

**核心抗诱惑原则**：任何任务功能新增前必须先回答 **"这能让打卡数据更有价值吗？"** 答不出来就拒绝。SPEC §1.3 + §5.5 "**不做**" 列表是底线。

**推翻条件**：
- 3 个月后度量 §11 显示打卡 × 任务关联率 < 10% → 联动失败，考虑砍掉任务模块或回退为方案 1
- 任务完成率 < 30% → 用户在拿任务当愿望清单，不是工作工具，重新设计或砍掉
- 任务表无止境膨胀（>500 条）且 done 比例极低 → 同上

## ADR-007 — 多 LLM provider 抽象，默认 DeepSeek

**日期**：2026-06-07

**背景**：用户希望支持第三方 LLM key，特别是国内的 DeepSeek。原本只想接 Claude。

**选项**：
1. 只接 Claude
2. 只接 DeepSeek
3. **抽象层 + 多 provider，OpenAI 兼容统一一个适配器**

**决定**：选 3。Provider trait 抽象，内置预设：Claude、DeepSeek、Kimi、通义、SiliconFlow、OpenRouter、Ollama、自定义 OpenAI 兼容。默认 DeepSeek。

**理由**：
- 国内主流 LLM 厂商全部提供 OpenAI 兼容端点，一个适配器覆盖 90% 场景
- 用户在国内，DeepSeek 价格 + 中文 + 网络都最合适
- Claude 留作高质量选项；本地 Ollama 留作隐私选项
- 抽象层带来的代码复杂度可控（trait + 3 个 impl）

**推翻条件**：
- 如果发现某个 provider 有特殊能力（如 Claude 的 prompt caching）值得专门优化，单独建 provider
- 如果用户从不切换 provider，未来可考虑简化为单 provider + 配置

## ADR-006 — STT 默认本地

**日期**：2026-06-07

**背景**：每小时打卡录音几秒到十几秒，量小但频次高。

**选项**：
1. 云端 Whisper API
2. **本地 faster-whisper / SenseVoice**
3. 系统自带语音识别（macOS Speech Framework）

**决定**：默认本地（SenseVoice Small 优先，中文最准），云端作为可选。

**理由**：
- Mac M1+ 跑 SenseVoice Small 实时率 < 0.1，零延迟体验
- 零成本、零隐私顾虑
- 一次性下载 ~250MB 模型，可接受
- macOS Speech Framework 中文质量差且要权限

**推翻条件**：用户机器太老跑不动；或者中文准确率不达标用户反馈。

## ADR-005 — 物品分类用"纯标签 + LLM 聚类建议"

**日期**：2026-06-07

**背景**：物品分类怎么定？预设分类树 vs 自由标签 vs 二者并存。

**选项**：
1. 预设几十个固定分类
2. **纯标签 + LLM 聚类**
3. 分类 + 标签并存

**决定**：选 2。物品不存"分类"字段，只存 tags。LLM 在录入时建议主标签；定期由 LLM 分析所有标签，建议升格为"主标签"。

**理由**：
- 预设分类永远不全：买个机械键盘是"电子"还是"办公"？
- 标签更灵活：一件物品可以同时是 #办公 #键盘 #送礼物
- LLM 帮助避免"标签爆炸"：每条录入只用既有标签或建议新的
- 数据上简单：events.tags 一个 JSON 数组搞定

**推翻条件**：标签数超过 200 仍无法收敛、用户找东西困难。

## ADR-004 — 万物皆 event

**日期**：2026-06-07

**背景**：物品和打卡是两类不同数据，要分表还是合并？

**选项**：
1. items / checkins / notes 三张表
2. **events 一张表 + type 字段 + JSON structured**

**决定**：选 2。

**理由**：
- 未来扩展新类型（书、电影、饭、人）零成本
- 时间轴回顾只需要一个查询
- structured 字段用 JSON 存，schema 演进无痛
- 缺点：每种 type 的字段约束在代码里，不在 schema 里——可接受（反正是单用户、自己管）

**推翻条件**：某类数据量极大（如打卡积累 5 年 ~ 40000 条），查询性能不达标。届时迁移到分表。

## ADR-003 — 只做 Mac，承认覆盖范围限制

**日期**：2026-06-07

**背景**：用户希望"每小时记录在干什么"，但不在 Mac 前的时间打卡入口缺失。

**选项**：
1. Mac App 主打物品库，打卡功能砍掉
2. Mac App + 手机轻量补录入口
3. **先只做 Mac，预留手机端 HTTP 接口**

**决定**：选 3。第一阶段只做 Mac，但后端启动 `127.0.0.1` HTTP server，将来手机端可调。诚实承认这一阶段"打卡只覆盖在电脑前的时间"。

**理由**：
- 一上来就做双端，摊子太大，3 个月做不完容易弃坑
- 先验证 Mac 端是否真在用——如果连 Mac 端都坚持不下来，做手机端也没意义
- 架构预留口子（HTTP API + 万物皆 event 表），将来扩手机端零迁移
- 用户接受这个权衡

**推翻条件**：3 个月后 Mac 端日活高、想扩到手机，按二阶段计划做 Android PWA 或快捷指令。

## ADR-002 — Tauri + Svelte + Rust，不用 Electron 或 Swift

**日期**：2026-06-07

**背景**：Mac App 技术选型。用户不熟任何桌面技术栈，主要靠 AI 写代码。

**选项**：
1. SwiftUI 原生 → 体验最好，但 Swift 调试难度大
2. Electron + React → 上手快，但包大（150MB+）、性能差
3. **Tauri + Svelte** → 包小（10-20MB）、性能好、菜单栏/通知/快捷键 API 完整
4. 菜单栏脚本（Hammerspoon） → 简陋

**决定**：选 3。

**理由**：
- 用户不会 Swift，调试卡住没出路
- 一个菜单栏小工具不该是 150MB
- Tauri 2 的 tray-icon、notification、global-shortcut 都是标准 plugin
- Svelte 比 React 模板更直觉，前端代码少
- Rust 后端写少量胶水代码即可（SQLite + HTTP + LLM 调用）

**推翻条件**：Tauri 2 某个核心 plugin 在 Mac 上踩到无法解决的 bug。

## ADR-001 — 不用 Telegram Bot

**日期**：2026-06-07

**背景**：最初方案是 Telegram Bot，但用户不用 Telegram；国内可选 IM 通道都有坑。

**选项**：
1. 企业微信自建应用 → 可主动推、能收语音，但 App 偶尔登出
2. 飞书机器人 → 同上，更工具感
3. ntfy/Bark + 网页表单 → 1 周能跑，但无对话历史
4. **Mac App** → 用户选择
5. Android 原生 App → 周期长

**决定**：选 4（Mac App）。

**理由**：用户明确表态。承认局限（见 ADR-003）。

**推翻条件**：用户改主意。
